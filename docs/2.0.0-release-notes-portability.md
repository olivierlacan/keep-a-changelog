# Release notes: host conformance and portability (research)

Background research for the 2.0.0 "Is a changelog the same as release notes?"
section and its lock-in argument. Two questions:

1. When projects publish release comms through a code host instead of a
   `CHANGELOG.md`, do those releases actually follow Keep a Changelog (KAC)
   recommendations?
2. Do hosts give you a simple way to port or export release notes back out?

**Observed: 2026-06-08.** Primary sources preferred (GitHub/GitLab REST APIs,
official host docs, the release `body`/`description` fields read live via
`gh api`). Quoted excerpts are verbatim. Gaps are flagged as gaps.

## 1. Do host-published releases follow KAC? (empirical, 18 projects)

Sampled the most recent one or two releases of 18 well-known projects that keep
no standard `CHANGELOG.md` and rely on host releases (or a `RELEASES.md`). Scored
against the 2.0 rubric: dated in the record, grouped by meaning, curated
user-facing prose (not a PR/commit dump), breaking changes marked, security
surfaced.

**Verdict: partially aligned at best, rarely fully.** Almost none satisfy the
whole rubric at once. Rough breakdown of the 18:

- **Genuinely curated and meaning-grouped (~4 to 5):** huggingface/transformers
  (groups by meaning, has an explicit `## Breaking changes` section flagged with
  an emoji, even records a yank: "v5.10.0 was yanked as we publish on a corrupted
  branch"); microsoft/PowerToys (curated per feature area with Added/Fixed verbs);
  rust-lang/rust `RELEASES.md` (`## Language` / `## Compiler` / `## Libraries` /
  `## Stabilized APIs`, human-written); astral-sh/ruff (dated in body,
  `### Bug fixes` / `### Rule changes`). facebook/react-native is close, even
  using the literal KAC type names `### Added` / `### Changed` / `### Fixed`, but
  carries `(commit by @user)` provenance lines and no in-body date.
- **Auto-generated PR or commit dumps (~6 to 7):** langchain
  ("release(core): 1.4.1 (#37922)" lines), next.js (`### Core Changes` +
  `### Credits`), polars (note its grouping includes `## Performance
  improvements`, exactly the type KAC says not to add), yt-dlp, deno, zed, the
  `## What's Changed` tail of ollama.
- **Freeform non-changelogs (~4 to 5), including the worst case, the empty
  redirect:** vscode's release body is literally just
  `https://code.visualstudio.com/updates/v1_123`; TypeScript points to a devblog
  post; bun redirects to its blog; tauri's sampled body is a raw `cargo audit`
  dump. The host "release" carries zero change content.

Two findings stand out:

- **Breaking changes are almost never marked distinctly.** transformers is the
  lone clear example in the sample; a few others list removals without a marker.
- **Security is essentially unmet.** Only one project led with a CVE
  (excalidraw v0.18.1: "addressing upstream Mermaid XSS vulnerability
  CVE-2025-54881 / GHSA-7rqq-prvp-x9jh").

**`RELEASES.md` location is not a proxy for quality.** rust's in-repo file is
curated and KAC-shaped in all but the `**Breaking:**` marker; deno's in-repo file
is the opposite: it prints the date the way KAC asks (`### 2.8.2 / 2026.06.03`)
but the entries are a near-raw `feat(...)`/`fix(...)` commit dump. File location
(in git vs. host DB) and curation quality are independent axes.

Caveats: one or two releases per project, so this reflects each project's current
template, not its whole history; several latest releases sampled were RC/beta;
offsite-redirect bodies were scored as the host artifact (empty), though the
offsite page may be curated. None of the 18 ships a `.github/release.yml`, so
where the "What's Changed" PR-list format appears it is invoked ad hoc.

## 2. Can you port or export release notes back out? (per host)

| Host | Notes stored as | Read API | CLI | One-click export to a file? | Writes an in-repo changelog? | Portability |
|---|---|---|---|---|---|---|
| **GitHub** | DB-backed release object on a tag; Markdown `body`. Not in `git clone`. | `GET /repos/{o}/{r}/releases[/latest,/{id}]`; GraphQL `releases` | `gh release list/view` | No | Only to a release object via `POST .../releases/generate-notes` (not a repo file). Config `.github/release.yml`. | Scriptable, not one-click |
| **GitLab** | DB-backed release object on a tag; Markdown `description`. Not in `git clone`. | `GET /projects/:id/releases[/:tag]` | `glab release` | No | **Yes, uniquely:** `POST /projects/:id/repository/changelog` commits a section to `CHANGELOG.md`, but built from **git commit trailers**, not from the Release objects. | Scriptable; the in-repo generator is the one host feature that puts notes back in git |
| **Gitea** | DB-backed release object on a tag; Markdown body. | `GET /repos/{o}/{r}/releases...` (GitHub-style) | `tea release list` | No | No | Scriptable, not one-click |
| **Forgejo / Codeberg** | Same as Gitea (API-compatible hard fork). | Gitea-compatible `/repos/{o}/{r}/releases` | Gitea-compatible | No | No | Scriptable, not one-click |
| **Bitbucket Cloud** | No releases feature at all; only tags, commits, Downloads. | None (no releases endpoint) | None | N/A | No | Nothing host-stored to export |
| **SourceHut** | Release notes **are the annotated git tag message** (`git tag -a`). | git itself; GraphQL exists | `hut` | N/A (already in git) | N/A (model already keeps notes in git) | **Most portable: `git clone` carries the notes** |

### Synthesis: is the page's lock-in claim fair?

**Yes, with one honest qualifier: exportable, but not portable by default.**

- The **default workflow does keep notes in the host's database**, attached to a
  tag, not as a file in the repository. A plain `git clone` (or a fork, or a
  mirror to another host) does **not** bring the release notes. That is the core
  of the KAC portability argument and it holds for GitHub, GitLab, Gitea, and
  Forgejo/Codeberg.
- **"Exportable" is true but is not "portable."** Getting the notes out needs the
  host's API or CLI plus a script, the host being online, and reassembling
  JSON-wrapped Markdown into one file. It is scriptable, not one-click: no major
  host ships a native "generate a `CHANGELOG.md` from all my releases" button.
- **The export is lossy.** Bodies carry host-specific auto-sections ("New
  Contributors", "Full Changelog" compare links), PR/label categories rather than
  the six types, and per-host field names. Concatenating them is usable but not a
  clean KAC document without editing.
- **Two honest exceptions** that the claim should not overreach against:
  1. **SourceHut inverts the model**: notes live in the annotated tag, so they do
     travel with `git clone`.
  2. **GitLab's changelog API** is the one host feature that writes notes into the
     repo, though from commit trailers, not from the host-stored release objects.

**Defensible wording:** "By default, host release notes live in the host's
database, not your repository, so they do not come with `git clone`. You can
export them through the host's API, but that is scripted, host-specific, and
lossy, not portable." Avoid an absolute "impossible to get out."

### Do the host generators reference KAC? No.

- **GitHub** auto-generated notes categorize by **PR labels** you define in
  `.github/release.yml` (e.g. `Semver-Major`), explicitly not the six KAC types.
- **GitLab**'s changelog generator categorizes by a **git commit trailer**
  (default `Changelog:`) whose allowed values are
  `added/fixed/changed/deprecated/removed/security/performance/other`. Six of
  those eight overlap KAC names, but it is GitLab's own trailer-keyed system
  (configured in `.gitlab/changelog_config.yml`) and the docs never mention KAC.

## 3. Timeline (for the "capture of release comms" narrative)

- **GitHub Releases launched 2013-07-02** ("Release Your Software,"
  github.blog/news-insights/product-news/release-your-software/). Releases pitched
  as "first-class objects with changelogs and binary assets."
- **Keep a Changelog founded 2014.** Predecessors: GNU `ChangeLog` and `NEWS`
  files (decades older).
- **GitLab Releases launched in GitLab 11.7 (January 2019).**
- **GitHub auto-generated release notes: public beta 2021-10-04**
  (github.blog/changelog/2021-10-04-improvements-to-github-releases-public-beta/).
  GA date not found in docs.
- **GitLab repository changelog API:** widely cited as GitLab 13.9 (Feb 2021) but
  **not confirmed** from primary docs (only later incremental history is
  documented); the POST endpoint went GA in 17.3.

The shape of the decade: hosts made release comms a first-class, DB-backed object
(2013, 2019), then automated it from PRs and commits (2021), which pulls the
narrative away from the in-repo file and into host-proprietary tooling. The
"rise of `CHANGELOG.md`" beyond the KAC-2014 anchor and the GNU precedent is not
precisely dated here; it would need a dedicated dig for a hard citation.

## 4. Third-party "releases to CHANGELOG" tools

- **github-changelog-generator** (github.com/github-changelog-generator/github-changelog-generator)
  reads FROM GitHub host data (tags, issues, labels, merged PRs); closest to a
  true host-releases-to-CHANGELOG exporter.
- **release-please** (github.com/googleapis/release-please): Conventional-Commits
  driven; writes both a `CHANGELOG.md` and GitHub Releases.
- **git-cliff** (github.com/orhun/git-cliff): generates `CHANGELOG.md` from git
  commit history; host-agnostic, does not read host release objects.
- **towncrier** (github.com/twisted/towncrier): builds from "news fragment" files
  committed in the repo.
- **conventional-changelog** (github.com/conventional-changelog/conventional-changelog):
  commit-message driven.

Note: most of these read from commits or repo files, not from host release
objects. Only github-changelog-generator genuinely pulls host release/issue data.
