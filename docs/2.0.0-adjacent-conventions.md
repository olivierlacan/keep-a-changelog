# Adjacent conventions in real-world Keep a Changelog adopters

**Audit date:** 2026-06-07
**Goal:** Surface conventions that well-known KAC adopters layer *on top of* the
Keep a Changelog spec, but which the KAC 2.0 page does not address — so the
maintainer can decide whether the spec should mention them.

## Method

Fetched the raw changelog of each project (curl for raw Markdown; WebFetch for
the Unity HTML page) and read enough of each to catalog every convention that
goes beyond what KAC 2.0 already covers. Every flagged convention is backed by a
quoted line from a named project. Prevalence is counted out of the 10 sampled
projects. Rich, Textual, httpx, and testfx are the most carefully maintained and
are weighted accordingly.

**Projects sampled (10):** microsoft/testfx, google/brotli, cloudflare/boringtun,
Unity com.unity.services.core, Textualize/rich, Textualize/textual, encode/httpx,
cncf/landscape2, OWASP/pytm, wikimedia/less.php.

Note: wikimedia/less.php keeps its log in `CHANGES.md`, not `CHANGELOG.md`, and
omits both ISO dates and reference links entirely — itself a small deviation.

## Most worth KAC's attention

1. **Contributor attribution.** 3 of 10 projects, and the two styles diverge
   sharply: GitHub's machine-generated `by @user in [#PR]` (testfx) versus a
   hand-written real name in parentheses (less.php: `(Timo Tijhof)`, brotli
   credits via a profile URL). This is the single most recurrent thing KAC is
   silent on. A short note acknowledging both styles — and the portability
   caveat that `@handle` is host-specific, mirroring the existing issue/PR-link
   caveat — would be high-value and low-controversy.

2. **Extra category headings beyond the six.** 6 of 10 projects add categories.
   The most common net-new one is brotli's `### Improved` (13 uses) for
   non-bug-fix optimizations; testfx adds `### Housekeeping` (12) and `### New`
   for chores; boringtun and pytm promote breaking changes to their own
   `### BREAKING`/`### Changes` heading. KAC currently presents the six as *the*
   set. A sentence on whether projects may add categories (and a caution to keep
   them few/consistent) would reflect real practice.

3. **Breaking-change presentation is inconsistent and worth pinning down.** KAC
   2.0 sanctions a `**Breaking:**` per-entry lead-in, but in the wild we see at
   least three forms: a dedicated `### BREAKING` section (pytm), an
   uppercase-bold `**BREAKING**:` entry prefix (landscape2), and a `Breaking:`
   prose lead (boringtun). pytm and httpx also attach *upgrade instructions* to
   breaking entries. Explicitly endorsing one canonical form — and noting that
   inline migration guidance is welcome — would reduce this drift.

4. **CVE IDs in Security entries** (brotli, testfx). KAC defines the Security
   category but says nothing about *what* a security entry should contain. Two
   of the most security-relevant projects both lead with a CVE ID. A one-line
   suggestion ("include a CVE ID and advisory link where one exists") is a
   natural, uncontroversial refinement of the existing Security guidance.

## Project-specific / niche (probably skip)

- **`### Artifacts` package lists** (testfx only) — useful for a multi-package
  NuGet release, but specific to that publishing model.
- **`### New Contributors`, `See full log` inline diff links, HTML anchors,
  `[Experimental]` labels** (all testfx only) — these are artifacts of GitHub's
  auto-generated release notes pasted into the changelog, not a deliberate KAC
  extension. Worth mentioning *only* if KAC expands its existing
  changelog-vs-release-notes section.
- **Sub-area prefixes** (`decoder:`, `java:` — brotli only) — already covered in
  spirit by the monorepo/per-component grouping guidance.
- **Per-release multi-paragraph prose** (httpx) — an expansion of the one-line
  summary KAC already permits; no new rule needed.
- **Decorative emoji** (one line in testfx) — too rare to standardize.
- **Deviations, not additions** (call out but don't bless): httpx's non-ISO
  `(6th December, 2024)` heading dates, less.php's missing dates / missing
  reference links / `CHANGES.md` filename. These contradict existing KAC rules
  rather than extending them.
