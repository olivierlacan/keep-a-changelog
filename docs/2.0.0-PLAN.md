# Keep a Changelog 2.0.0 — Release Plan

> Working planning document for the 2.0.0 release.
> Status: **draft**. Owner: @olivierlacan. Last reviewed: 2026-06-04.

## Why a 2.0

Keep a Changelog shipped its first version in 2014. In the decade since, the
way software is released changed underneath it:

- **Continuous delivery / SaaS** made discrete, numbered releases optional. Many
  projects ship many times a day and never cut a "version" a human reads.
- **Conventional Commits + automation** (`semantic-release`, `release-please`,
  Changesets, `git-cliff`) industrialized changelog *generation* from commits —
  exactly the "git log dump" KAC was created to discourage.
- **Versioning pluralism**: SemVer is no longer assumed. CalVer, ZeroVer, and
  date-based schemes are common.
- **GitHub Releases** matured from a thin feature into a primary channel.
- **Dependency bots** (Dependabot, Renovate) generate enormous volumes of
  low-notability change.
- **LLMs now write and summarize code**, which makes "for humans, not machines"
  both more fragile (easy to auto-dump) and more important (a curated,
  human-readable record of *what actually changed* is a scarce, valuable thing).

2.0 is **not** "changes for change's sake." It exists to (a) update KAC's stated
opinions to this reality, (b) finally answer the handful of questions the
community has re-asked for ten years, and (c) state the principles that keep
changelogs human-curated in an AI-assisted world — without turning KAC into a
tool or a runtime.

**Notability filter for every proposed change:** does it resolve a recurring,
high-engagement community question, *or* reckon with a real shift in release
practice since 2014? If neither, it's out.

## What the 2.0 draft already does

The current `source/en/2.0.0/index.html.haml` is a **tone-and-framing refresh**,
not a structural one. It already:

- Reframes a changelog as a *document* about *iterations* of a project.
- Generalizes "Semantic Versioning" → **"versioning schemes"** (`data.links.schemes`).
- Adds the **core thesis**: *"Making changes and communicating about changes are
  two fundamentally different things,"* and explicitly rejects auto-categorizing
  every commit.
- Adds/rewrites FAQs: filename pedantry, a more generous **GitHub Releases** take
  (with a portability caveat), a warmer **"can changelogs be parsed?"** answer
  framed around upgrade filtering, and a **"What's a git log?"** explainer.
- Ships **`new` / `updated` CSS classes** (PR #600) so translators can see what
  changed; PR #691's coverage tool measures the resulting translation debt.

It does **not** yet address: breaking changes, change-type questions,
SaaS/continuous delivery, Conventional Commits positioning, large-changelog
hygiene, per-release summaries, or the LLM angle. That is the work below.

## Decisions locked (2026-06-04)

1. **Breaking changes** → resolve via **convention + SemVer mapping**, not a new
   section. (See Tier 1, item 1.)
2. **Change types** → hold at six. The types answer *"what kind of change?"*; the
   *reason* (improved, optimized, refactored, performance, dependency bump) lives
   in the entry **text**, never as a new type. The real gap to close is the
   **Changed / Fixed / Security** boundary. (See Tier 3.)
3. **Tooling/LLM layer** → state principles in 2.0; any tools we offer must be
   **universal and vendor-neutral** (no GitHub-Action-specific deliverables). The
   portable artifact is a precise *format specification*, not a vendor integration.
   "JSON Schema" is parked as an open question (it validates JSON, not Markdown —
   wrong primitive unless we also define a `changelog.json` projection).
4. **Packaging** → this document.

## Tier 1 — Definitely in 2.0

High demand × philosophically core × long-unresolved.

### 1. Answer "how do I denote breaking changes?" — #480 (35 comments, open since 2014)

The single most-debated thread in the repo. Resolution:

- **No `Breaking` section** — it duplicates `Removed`/`Changed` and fragments the
  six types.
- Recommend a **convention**: a `**Breaking:**` lead-in on the relevant
  `Changed` / `Removed` entries.
- State plainly that **a major version bump implies breaking changes**, and that
  the changelog's job is to make the upgrade pain obvious (ties to the existing
  "Ignoring Deprecations" anti-pattern).
- Fold in #536 / #698: **say which interface SemVer covers** for your project
  (CLI, library API, wire/RPC, file formats, GUI). "Breaking" is meaningless
  until the contract is named.

```
## [2.0.0] - 2026-06-04
### Changed
- **Breaking:** `parse()` now returns a `Result` instead of throwing
### Removed
- **Breaking:** dropped Node 14 support
```

*Translation impact: additive FAQ — cheap. Resolves the repo's #1 thread.*

### 2. Distinguish changelogs from release notes / public vs internal — #463 (14 comments)

Promote the draft's seed into a full FAQ. KAC is **for humans reading the repo**;
release notes are a **derived, audience-specific view**. This is also the
principled answer to "public vs internal" (#463): you don't need an `[internal]`
type — you curate notable, outward-facing changes, and *generate* a narrower
release-notes view from them when needed.

### 3. Finish the versioning-scheme generalization

The draft swapped "Semantic Versioning" → "versioning schemes." Complete the
thought: explicitly acknowledge **CalVer, date-based, and continuous** schemes,
and how "Unreleased" and dated entries work when there is no discrete version
number (SaaS / continuous delivery). Reckons directly with the post-2014 shift.

### 4. Position KAC relative to Conventional Commits & automation — #611 (▲3)

The decade's biggest ecosystem change. A clear FAQ:

- Conventional Commits / `semantic-release` / `release-please` / Changesets /
  `git-cliff` operate on **commits** (the input).
- KAC describes the **human-facing output**.
- They are **complementary**: a generated changelog is a *draft*, not a finished
  one — curation is the step that makes it "for humans."

This is the natural extension of the draft's "changes ≠ communicating about
changes" thesis, and it's the bridge to the AI section below.

## Tier 2 — Strongly consider (needs a final call)

### 5. Optional per-release summary line — #597 (▲5, highest upvotes)

Bless an optional intro sentence/paragraph under a version heading, before the
typed sections — a one-line narrative of what the release is *about*. Cheap,
popular, doesn't touch the type system.

### 6. Large / long-running changelog hygiene — #652 (▲3), #529, #565

No current guidance. Recommend: keep **one file** (Rust's ~16k-line, Ctrl-F-able
changelog as precedent), with *optional* archiving by year or major version and
clear linking. Counsel against pruning history.

### 7. Linking to issues / PRs — #612, #522

A short convention: link sparingly, prefer human prose over a wall of `(#1234)`.

### 8. Monorepos & continuous delivery — #510, #526 (9 comments)

Guidance: per-package `CHANGELOG.md` for monorepos; date-stamped entries when
there is no discrete version.

## Tier 3 — Hold the line, but answer once and for all

### 9. "Why aren't there more change types?" — #490 (14c), #518, #532, #481 (15c)

Recurring requests: Improved, Optimized, Refactored, Performance, Dependencies,
Known Issues. **Decision: the canonical set stays at six.** The FAQ frames it as
**what vs. why**:

- The six types answer **"what kind of change is this?"** — the *what*.
- *Improved, Optimized, Refactored, Performance* are **reasons / details**, not
  kinds of change. They belong in the entry **text**, e.g.
  `- Rewrote the JSON parser; ~3× faster on large files` under `Changed`.
- **Dependencies** is not a human-facing kind of change at all — a bump may be
  innocuous, breaking, or a fix. If it's notable to a human, describe its
  *effect* in the relevant type; if it isn't, it doesn't belong in a changelog.
- **Known Issues** (#481) is a *discovery*, not a *change*. Append it to the
  affected release or keep it in the issue tracker; when fixed, it lands under
  `Fixed`.

Adding types would push the burden back toward "categorize every commit" — the
exact tedium 2.0 argues against.

### 10. Clarify the Changed / Fixed / Security boundary (the actual confusion)

The genuine ambiguity isn't "we need more types" — it's that people can't tell
**Changed vs Fixed vs Security** apart. 2.0 should give a crisp rule of thumb:

- **Fixed** — behavior that was *wrong* (vs. documented/intended) is now correct.
- **Changed** — behavior that was *working as intended* now works differently.
- **Security** — a fix or change that addresses a *vulnerability*; called out
  separately because the audience and urgency differ.

Worth a short worked example or two, since this is where most "which section?"
questions (#569, #588, #524) actually come from.

## The human + LLM layer (principles in 2.0; tools, if any, must be universal)

KAC isn't software, and 2.0 won't make it one. But it can **state principles**
and, later, **offer portable artifacts** that keep changelogs human-curated as
LLMs write more code.

### In 2.0 (text only)

- Make explicit that **machines draft, humans curate** — and that this matters
  *more*, not less, when an LLM can emit a plausible changelog from a diff in
  seconds. The risk isn't a missing changelog; it's a fluent, auto-generated one
  that no human ever read.
- A short, **portable** "writing changelog entries from a diff" guidance aimed at
  both humans and AI assistants: summarize *notable, human-facing* changes; never
  paste a git log; map to the six types; put the *why* in the text; flag breaking
  changes; omit non-notable churn. This is the literal antidote to *"let your
  enemies turn your changelogs into git commits."*

### Open questions to settle before building anything

- **What is the universal artifact?** The portable, vendor-neutral contribution
  is a **precise format specification** — heading levels, section names and
  ordering, the `YYYY-MM-DD` date rule, `[YANKED]`, link-reference style — plus a
  **conformance corpus** (example changelogs marked valid/invalid). Any linter in
  any language can build against that. We don't ship a linter tied to one CI.
- **What would "JSON Schema" actually buy us?** A JSON Schema validates *JSON*,
  not Markdown — so on its own it's the wrong primitive. It only becomes useful
  if we also define a canonical `changelog.json` **projection** of a parsed
  changelog (which tools/LLMs could read and write reliably). Decide *whether we
  want that projection at all* before investing in a schema for it.
- **Compliance badge (#515, 10 comments):** desirable, but only meaningful once
  "valid" is defined by the spec above. Sequence it after the spec, and keep it
  generatable without any single vendor.

**Sequencing:** 2.0 states the principles. The format spec + conformance corpus
is a *separate, post-2.0* deliverable. No vendor-specific tooling.

## Translation impact (PR #600 + #691)

Every reworded sentence silently invalidates the 0.3 / 1.0 / 1.1 translations.
The `new` / `updated` classes (#600) and the coverage tool (#691) exist to manage
this. Two consequences for how we sequence 2.0:

- **Prefer additive content (new FAQ sections) over rewording existing
  sentences.** Additive text is cheap to *append* in each translation; reworded
  text must be *re-verified* line by line.
- **Batch all wording changes into 2.0 and mark them**, then drive the
  translation backlog from the coverage tool. Structural/normative changes (new
  conventions) deserve more translator attention than prose polish — ideally the
  tool can weight those.

Most Tier 1–3 items above are **additive FAQs**, which is deliberate: it
maximizes reader value per unit of translation debt.

## Explicit non-goals for 2.0 (considered, deferred)

State these in the release so the community sees they were weighed, not forgotten:

- A localization platform (#539).
- Expanding the canonical change-type set (Tier 3 decision).
- Deep monorepo *tooling* (guidance yes, tooling no).
- Hosting a changelog registry or a tool directory.
- *Mandating* any machine format. KAC stays a convention, not a runtime.
- Any single-vendor integration (e.g. a GitHub-Action-only validator).
