# Keep a Changelog 2.0.0 — restructure outline (proposal)

Goal: move away from an 18-entry FAQ toward leaner, integrated guidelines that
read like a manifesto. Keep a short FAQ only for genuine "what about my
situation?" edge cases. Pull English-/author-specific material into a standalone
**References** block that isn't translated. Reduce first-person prominence without
losing the human voice.

> Timing note: there are **no 2.0 translations yet**, so this restructure costs
> zero translation rework. It's the free moment to do it.

## New top-level shape (page order)

1. **Header** — title, tagline, live `CHANGELOG.md` example. *(unchanged)*
2. **What a changelog is** — short intro prose.
3. **How to keep a changelog** — the integrated core (prose + a few sub-headings).
4. **What makes a changelog worse** — the antipatterns.
5. **Changelogs, automation & LLMs** — one cohesive narrative.
6. **Miscellaneous** — things that don't fit elsewhere (no FAQ framing).
7. **About Keep a Changelog** — positioning, non-goals, contributing.
8. **References** — a translated section, kept deliberately lean.

## Decisions (locked)

1. The `#git` explainer stays as a **body aside**, and adds a note that these
   guidelines aren't limited to git — they apply to any version control system.
2. **Drop the `.new` / `.updated` markers** for 2.0; treat the page as a fresh
   translation baseline.
3. **No FAQ.** Leftover edge cases go into a **Miscellaneous** section.
4. **References is translated** (changed from standalone), but **synthesized more
   tightly** than other sections to keep the translation load light.

## Section detail

### 2. What a changelog is
Absorbs `#what`, `#why`, `#who`. Already prose; keep, lightly tightened.

### 3. How to keep a changelog
The big integration. One section, sub-headings:

- **Guiding principles** — the list (`#principles`).
- **Types of changes** — the six (`#types`), then woven in as prose:
  - picking between **Changed / Fixed / Security** (was `#which-type`);
  - why the set stays at six — the *type* is the *what*, the reason goes in the
    wording; Dependencies and Known Issues answered (was `#more-types`);
  - flagging **breaking** changes with a `**Breaking:**` marker and naming the
    interface you treat as stable (was `#breaking`).
- **Structuring a release** — `Unreleased` at the top; a dated
  `## [x.y.z] - YYYY-MM-DD` heading; an optional one-line summary; then the typed
  sections; note your versioning scheme (absorbs `#summary`, `#versioning`, the
  `Unreleased` tip from `#effort`, and the date-format pointer).
- **Curate, don't dump** — the philosophy: communicating about changes is its own
  discipline; notable vs. non-notable; don't auto-categorize commits; "don't let
  your enemies turn changelogs into git commits" (the `#effort` thesis).

### 4. What makes a changelog worse
Absorbs `#bad-practices`, `#log-diffs`, `#ignoring-deprecations`,
`#confusing-dates`, `#inconsistent-changes`. Already prose; keep, maybe retitle.

### 5. Changelogs, automation & LLMs
One narrative absorbing four FAQs, in this flow:
- a consistent changelog is readable by tools and models; there's **no separate
  machine format**, and there won't be — this page is the format (`#automatic`);
- **machines can draft, humans curate**; the failure mode; the brief; drop it in
  an `AGENTS.md` (`#llms`);
- **Conventional Commits** and generators are complementary; a generated changelog
  is a draft (`#conventional-commits`, condensed);
- **CI** does the mechanics, never a hard gate (`#ci`).

### 6. Miscellaneous
A catch-all for guidance that doesn't fit the narrative sections — written as
short prose entries, not as a Q&A:
- What should the file be named? (`#filename`)
- Changelog vs. release notes and release posts? (merges `#release-notes` + `#releases`)
- What about yanked releases? (`#yanked`)
- Should you ever rewrite a changelog? (`#rewrite`)
- What if it gets too big? (`#large-changelog`)
- What about monorepos? (`#monorepos`)
- Should I link to issues or pull requests? (`#linking`)

### 7. About Keep a Changelog
Absorbs `#standard` (positioning), `#scope` (non-goals), `#contribute` (with
softened first person).

### 8. References (translated, but kept lean)
- The Changelog podcast appearance (`#press`) — reframed as a credit, not
  "I went on…".
- The "better changelog convention" link.
- Synthesize tightly: a sentence or two per item, so it's the lightest section to
  translate.

The `#git` "what's a git log?" explainer stays as a **body aside** (per decision 1),
with an added note that the guidelines apply to any version control system, not
just git.

## Voice

Keep the human voice, reduce first person. Soften `#effort`'s "tedium I wouldn't
wish on anyone" → "tedium no maintainer should endure"; the podcast "I went on…"
becomes a credit in References.

## Translation & tooling implications

- Free now (no 2.0 translations exist).
- After the restructure, a section-by-section 1.1→2.0 diff is messy (sections
  merge/rename), so 2.0 is cleaner treated as a **fresh translation baseline**.
- The `.new` / `.updated` markers lose meaning when nearly everything is new vs
  1.1; options: mark only whole-new sections, or drop per-paragraph markers for
  2.0 and rely on "translate the 2.0 page" as the instruction. The
  migration/consistency tools still run; the annotation check just becomes less
  informative.
- References permanently leaves the translation surface.

## Net effect

~18 FAQ entries → ~7 edge-case FAQs + integrated prose. The page reads as a
manifesto with recommendations, not a Q&A.
