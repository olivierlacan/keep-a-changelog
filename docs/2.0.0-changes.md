# Keep a Changelog 2.0.0 — what changed since 1.1.0

The detailed record of every change between the 1.1.0 and 2.0.0 manifesto. The
2.0.0 entry in `CHANGELOG.md` is a synthesized, curated version of this document —
this is the long form for reviewers and translators.

## The most important thing first

**Nothing you have already written becomes invalid.** 2.0.0 adds guidance and
clarifies wording; it does not change the format. Unchanged:

- the six change types — `Added`, `Changed`, `Deprecated`, `Removed`, `Fixed`,
  `Security`;
- the `YYYY-MM-DD` date format and the `## [version] - date` heading;
- `[Unreleased]` and `[YANKED]`;
- reverse-chronological order and links to versions and sections.

There is no migration to perform on your changelog. This is an *upgrade guide* in
the sense of "what's new to know," not "what you must change."

- **14 new sections** of guidance.
- **11 sections revised** (wording, mostly held back from minor releases — see the

## New guidance

### Working with the change types

- **`#which-type` — Changed, Fixed, or Security?** A rule of thumb plus a
  "was the old behavior a bug?" test for the boundary people ask about most.
- **`#more-types` — why there aren't more types.** The type is the *what*; the
  reason (improved, optimized, refactored, performance, dependency bump) goes in
  the entry's wording. Direct answers for Dependencies and Known Issues.
- **`#breaking` — marking breaking changes.** A `**Breaking:**` marker on
  `Changed` / `Removed` entries rather than a new section, plus: name which
  interface your versioning scheme treats as stable.

### Versioning and release structure

- **`#versioning` — you don't have to use Semantic Versioning.** CalVer, a plain
  number, or a date are fine; continuously deployed projects can keep dated
  entries under `Unreleased` with no version number.
- **`#release-notes` — changelog vs. release notes.** Different audiences and
  homes; derive release notes from the changelog rather than keeping two records.
  Also the clean answer to "internal vs. public" changes.
- **`#summary` — optional per-release summary.** A short, high-level note at the
  top of a version is welcome but never required.

### Automation and the LLM era

- **`#llms` — LLM-generated changelogs.** "Machines can draft, but humans curate";
  the failure mode is now a fluent changelog no one read. Includes a portable
  brief to drop into an `AGENTS.md` / `CLAUDE.md`.
- **`#conventional-commits` — relationship to Conventional Commits.** Commits are
  the input, the changelog is the human-facing output; a generated changelog is a
  draft that still needs curation.
- **`#ci` — fitting a changelog into CI/CD.** CI can do the mechanics (release-time
  `Unreleased`→version move, format checks, gentle reminders); it can't judge
  whether an entry is worth reading, so don't hard-gate every pull request.

### Maintaining a changelog over time

- **`#linking` — linking to issues or pull requests.** Reasonable and at your
  discretion, but URLs 404 when repos move and PR/MR numbers aren't portable;
  git tags and commit refs are repo-canonical.
- **`#large-changelog` — very large changelogs.** One file is usually fine; if you
  split, archive only stable-old versions and link both ways. Don't prune.
- **`#monorepos` — monorepos.** Unrelated projects → independent changelogs; a
  unified product → per-component changelogs *plus* a central one synthesizing
  whole-project changes.

### Scope and framing

- **`#scope` — what Keep a Changelog won't do.** The non-goals: no growing the type
  set, no machine format/schema/required layout, no becoming a tool or runtime, no
  single-vendor lock-in.
- **`#git` — what a "git log" is.** A short explainer behind the tagline.

## Revised sections

| Section | What changed |
|---|---|
| `#what` | A changelog is a "document" (was "file"). |
| `#why` | Notable changes between each "iteration" (was "release (or version)"). |
| `#principles` | Clearer phrasing throughout; "Semantic Versioning" generalized to "versioning schemes." |
| `#effort` | Expanded with the core thesis — communicating about changes is its own discipline; *making* changes and *communicating* about them are different; don't auto-categorize every commit ("don't let your enemies turn your changelogs into git commits"). |
| `#inconsistent-changes` | Minor wording ("…at all"); the "help collect antipatterns" aside folded in. |
| `#standard` | Reframed as historical and humbler — KAC doesn't aim to be the one true standard; broadened to any project doing outward-facing communication. |
| `#filename` | Added a rebuttal to the "a CHANGELOG should log *all* changes" pedantry. |
| `#releases` | Renamed from "GitHub Releases" and generalized to any host's release posts; reframed their value as limited and non-portable; adds "focus on the changelog file first, derive posts." |
| `#automatic` | Removed the Vandamme reference; added upgrade-filtering framing and the explicit no-machine-format stance (the manifesto is the format). |
| `#rewrite` | Adds guidance to note the date a changelog entry was updated after release. |
| `#contribute` | Humbler framing; a reflection on the decade since launch. |
