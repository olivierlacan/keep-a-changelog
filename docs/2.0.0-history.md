# Keep a Changelog — history claims, verified

Fact-check of a Gemini-generated "history of Keep a Changelog" before deciding
whether any of it belongs in the 2.0.0 release. Gemini fabricates and overstates,
so every factual claim here was checked against a primary source, and claims that
conflict with the project's settled editorial positions are flagged.

**Verified: 2026-06-07.** Method: this repo's `CHANGELOG.md`, `config.rb`, git
tags, and the GitHub REST API for repo facts; the live `keepachangelog.com` spec
for wording; the actual WP Tavern article; npm/crates registries for tooling
dates; web search for the Linux Foundation / CNCF and Linear/Beamer claims.
Confirmed adoption facts are reused from `docs/2.0.0-adoption-research.md` and
`docs/2.0.0-adoption-sectors.md` rather than re-verified.

Statuses: **VERIFIED** (primary source confirms), **UNVERIFIED** (could not
confirm), **DEBUNKED** (false/overstated), **CONFLICT** (contradicts a settled
project position).

## Verified timeline (only what holds up)

Each line has a primary source and a date.

- **2014-05-31 — Repository created by Olivier Lacan.** GitHub API
  `created_at: 2014-05-31T05:23:54Z`, `owner: olivierlacan`; matches this repo's
  `CHANGELOG.md` `[0.0.1] - 2014-05-31` and `docs/2.0.0-adoption-research.md`
  ("Repo created 2014-05-31"). The 0.0.1 entry: *"This CHANGELOG file to hopefully
  serve as an evolving example of a standardized open source project CHANGELOG."*
- **2015-02-16 — First notable press coverage (WP Tavern), by Sarah Gooding.**
  Title: *"Keep a CHANGELOG Project Aims to Standardize Best Practices for Writing
  Change Logs."* Describes "a site and corresponding GitHub repository" created by
  Olivier Lacan, "a software engineer at Code School," offering "an extensive
  collection of recommendations for writing better change logs."
- **2017-06-20 — v1.0.0 released.** `CHANGELOG.md` `[1.0.0] - 2017-06-20`;
  confirmed by git tag `v1.0.0`. This release added the Why/Who/How and Guiding
  Principles sections and a large batch of translations.
- **2019-02-15 — v1.1.0 released.** `CHANGELOG.md` `[1.1.0] - 2019-02-15`; tag
  `v1.1.0`.
- **2023-03-05 — v1.1.1 released.** `CHANGELOG.md` `[1.1.1] - 2023-03-05`; tag
  `v1.1.1`. (CHANGELOG notes "26 translations to date" at this point.)
- **Core principle — "Changelogs are for humans, not machines."** Live spec at
  keepachangelog.com/en/1.1.0 lists it verbatim as a Guiding Principle. (The
  Guiding Principles section was introduced in v1.0.0, 2017.)
- **Six change types — Added, Changed, Deprecated, Removed, Fixed, Security.**
  Confirmed verbatim and in this order on the live 1.1.0 spec. The 2.0.0
  CHANGELOG entry explicitly keeps them unchanged.
- **28 translations.** `config.rb` `$languages` / `$language_count` = 28 (primary,
  in-tree); see `docs/2.0.0-adoption-research.md`.

## Conflicts with project positions

1. **"The Manifesto" framing — CONFLICT.** The maintainer deliberately avoids
   "manifesto" and frames KAC as a *convention / guidelines grounded in existing
   good practice*, not something invented. The WP Tavern source agrees in spirit:
   it calls KAC "an extensive collection of recommendations," not a manifesto. Drop
   any "manifesto" framing.

2. **"Conventional Commits / git-cliff are complementary" — CONFLICT.** The
   project's just-codified stance is that a commit and a changelog entry are
   written for *different readers*; commit-driven generators are not a KAC-aligned
   partner. Gemini's "the structure of KAC without the manual labor" framing
   directly contradicts this. The tooling *dates* are fine to state; the
   "complementary" claim is not. (Note also: `conventional-changelog` predates
   KAC by ~5 months — Jan 2014 vs May 2014 — so framing it as built on KAC's
   structure is also chronologically wrong.)

3. **"Over 30 languages" — CONFLICT / DEBUNKED.** The defensible, primary-source
   count is **28** (`config.rb` `$language_count`). The site colloquially says
   "~30"; "over 30" is not supportable. Use 28, or the vaguer-but-safe "dozens"
   already used in the 2.0.0 CHANGELOG.

4. **"Resident of Seattle" / personal details — CONFLICT (privacy/out-of-scope).**
   Personal details about the maintainer are out of scope for the page and are not
   treated as verified here. Omit.

## Recommendation

**Yes — there is enough verified, on-message material for a short history note in
2.0,** but only a small, conservative set: the 2014 origin, the human-readable
core principle, the 2017 v1.0.0 standardization of the six change types, and the
present-day reach (28 languages; tens of thousands of open-source changelogs, per
the adoption research). Everything else Gemini offered is either a framing
conflict (manifesto, CC-complementary), an overstatement (30+ languages,
Linear/Beamer), or unverifiable (LF/CNCF endorsement) and should be left out.

Draft note, in KAC's plain, humble, non-"manifesto" voice, using only verified
facts:

> Keep a Changelog began in 2014 as a small set of recommendations for writing
> changelogs people could actually read — built on existing good practice rather
> than invented from scratch. Its guiding idea has not changed: changelogs are
> for humans, not machines. Version 1.0.0 in 2017 settled the six change types —
> Added, Changed, Deprecated, Removed, Fixed, and Security — that the format still
> uses today. Since then the convention has been translated into 28 languages and
> adopted by tens of thousands of open-source projects.

(If a tighter version is preferred, the first and last sentences alone carry the
on-message essentials.)
